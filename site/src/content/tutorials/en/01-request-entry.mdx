Every agent system has to answer an unglamorous question before it gets to be clever: who catches the request? The sentence you type in the frontend has to first become a **run** — something with an identity, a lifecycle, and a stream you can watch — before the model gets to say a word. This stop is about how DeerFlow catches that sentence, files it as a run, installs the runtime context, and streams events back to you as they happen.

Hold onto one idea; everything else hangs off it: **DeerFlow has to be a runtime before it can be an agent.** The Gateway is that runtime host — it owns intake, the life and death of a run, the context it installs, the events it streams out, and knowing when the whole thing is done. The actual "thinking" doesn't start until the next stop.

## Two "runtimes," don't conflate them

The easiest trap here is collapsing three different things into one:

```text
LangGraph         = the execution kernel agents use (how a graph runs)
LangGraph Server  = the optional official HTTP runtime
DeerFlow Gateway  = the current primary HTTP runtime, LangGraph-API compatible
```

`backend/langgraph.json` still registers `lead_agent` — that keeps the official LangGraph Server / Studio / CLI working. But **on the main path, the Gateway imports and calls the graph factory directly**, no official server in between. So the path you actually run in production is the Gateway's own runtime, not LangGraph Server.

<MarginNote>One line to keep them straight: LangGraph runs the graph; the Gateway turns a request into an observable, rollback-able, billable run. Kernel vs. host.</MarginNote>

## The four parts that land a request

Open <SourceRef file="backend/app/gateway/routers/thread_runs.py" sha="0fb18e36" /> and <SourceRef file="backend/app/gateway/services.py" sha="0fb18e36" /> and you'll keep meeting four roles. Don't memorize the fields — remember which question each one answers.

**`RunCreateRequest`** — what a request looks like. It carries `input`, `config`, `context`, `metadata`, `stream_mode`, plus control bits like `interrupt_before / interrupt_after`, `multitask_strategy`, and `on_disconnect`. It's the full declaration of "how this run wants to run."

**`RunManager`** — the registry office for runs. `create_or_reject()` decides whether to file this run at all (e.g. the same thread is already running, so reject per `multitask_strategy`), then handles `set_status` / `cancel` / persisting run metadata.

**`StreamBridge`** — a decoupling gate. The background graph produces events; the HTTP side consumes them; the two never hold hands directly. `run_agent()` publishes into the bridge, `sse_consumer()` reads from it and formats SSE events. If the frontend disconnects, the back doesn't collapse with it.

**`run_agent()`** — the background worker that actually does the work (<SourceRef file="backend/packages/harness/deerflow/runtime/runs/worker.py" sha="0fb18e36" />). Note carefully: **it is not the agent. It's the hand that holds the agent while it runs.**

## `configurable`, or `context`?

`build_run_config()` does a translation job: it turns the fields in the HTTP request into the `RunnableConfig` shape LangGraph understands. And here sits the one piece of compatibility debt most worth remembering:

```text
config["configurable"]   the older runtime-options channel
config["context"]        the newer LangGraph runtime-context channel
```

In compatibility mode, DeerFlow writes the key fields into **both** — whitelisted fields like `model_name`, `mode`, `thinking_enabled`, `reasoning_effort`, `is_plan_mode`, `subagent_enabled`, `agent_name`, `is_bootstrap` each get a copy in either channel. Why the apparent waste? Because older DeerFlow code reads from `configurable` and newer `ToolRuntime.context` reads from `context`; write only one side and a whole class of consumers comes up empty.

Then `run_agent()` builds a LangGraph `Runtime`, packs run-scoped data (`thread_id / run_id / user_id / app_config`) into `runtime.context`, and tucks it back into `config["configurable"]["__pregel_runtime"]` so downstream middleware and tools can all reach it through LangGraph's runtime API.

<Philosophy title="Design rationale">

Why does the Gateway reimplement a LangGraph-compatible API instead of just using the official LangGraph Server?

Because that's how DeerFlow gets to **own** the application-level concerns: auth, thread/run metadata, the run event store, the SSE bridge, rollback, frontend compatibility, hot-reload boundaries, its own persistence. None of these are "how the graph runs" — they're "how a product serves the graph."

The cost is a layer of compatibility glue: request fields must be translated into LangGraph-flavored config, stream modes mapped, runtime context installed by hand. The lesson rhymes with the next stop — **funnel the compatibility awkwardness into the entry layer, and let the graph execution below stay clean.** Kernel runs; host absorbs the compatibility.

</Philosophy>

## Where it will trip you up

- **`configurable` and `context` are historical compatibility debt.** A field written to only one side may read empty for old code or for newer `ToolRuntime.context` consumers. Adding a field? First decide which whitelist it belongs to and whether it needs the dual-write.
- **`run_agent()` is an orchestration-heavy function.** It mixes graph invocation, persistence, stream publishing, rollback, tracing, and cleanup — powerful, but dense. Know which part you're touching before you touch it.
- **It runs as a background task.** Disconnect, cancellation, rollback, and final checkpoint serialization all need care, or you may return "half-finished state" as success.
- **It doesn't mutate `ThreadState` directly.** Run records, thread metadata, and stream events are written by the Gateway; the agent's graph state is written by graph execution. When state changes, first sort out whose hand moved it.
