The first time you open `build_middlewares()`, it looks harmless — just a pile of objects getting appended to a list. But the *arrangement* of that list decides the runtime behavior of the entire run that follows. The user's prompt doesn't go straight to the model; it travels through layer after layer of middleware first. They prepare the thread's file paths, handle uploaded attachments, set up the sandbox's lifecycle, inject the date and memory, and patch the message protocol the provider expects — so that by the time the model is actually called, the request has already been shaped into what DeerFlow wants it to see.

The last stop looked at how tools get collected and exposed. This one goes deeper: **how a single run crosses the middleware onion and finally reaches the model call.** The other half — how `after_model`, `wrap_tool_call`, and `after_agent` take the result back out in reverse order — is the next stop.

<MentalModel>Middleware is not a pipeline — it's an onion wrapped around the execution boundary: outer layers fire first on the way in, inner layers fire first on the way out.</MentalModel>

## One list, built in two passes

The assembly entry point is <SourceRef file="backend/packages/harness/deerflow/agents/lead_agent/agent.py" sha="d2cc991d" symbol="build_middlewares" />. Logically it works in two passes: first take a runtime base shared by the lead and its subagents, then stack the lead-agent-only middleware on top.

The shared base comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/tool_error_handling_middleware.py" sha="d2cc991d" symbol="build_lead_runtime_middlewares" />:

```text
shared base (reused by lead and subagent)
  ToolOutputBudget    spill a big tool output to a file, leave only a preview + path
  ThreadData          compute thread_data paths for workspace/uploads/outputs
  Uploads             surface uploaded files to the model (lead only)
  Sandbox             manage the sandbox lifecycle and sandbox_id persistence
  DanglingToolCall    repair dangling tool-call protocol in the history
  LLMErrorHandling    retry / trip the breaker / fall back when a model call fails
  Guardrail?          optional runtime authorization (only with a provider configured)
  SandboxAudit        the pass/warn/block gate for bash commands
  ToolErrorHandling   turn a tool exception into ToolMessage(status="error")
```

Then `build_middlewares()` appends the lead-only segment:

```text
lead-only segment (?-marked layers are conditional)
  DynamicContext      inject date and memory without touching the system prompt
  SkillActivation     on /skill, inject the full SKILL.md into this turn's request
  Summarization?      compress old messages when history grows too long
  TodoList?           plan mode's task list and completion reminders
  TokenUsage?         record token usage, attribute actions to product semantics
  Title               generate a title after the first full exchange
  Memory              queue the conversation into the memory queue when the run ends
  ViewImage?          when the model supports vision, surface viewed images
  DeferredToolFilter? hide unpromoted deferred tool schemas, reject illegal calls
  SubagentLimit?      cap parallel task/subagent count
  LoopDetection?      detect and break repetitive tool-call loops
  (custom middleware)
  SafetyFinishReason? clear untrustworthy tool_calls after a provider safety stop
  Clarification       turn ask_clarification into a run-level pause, always last
```

Two things are worth pinning down before reading on.

**First, order itself is a runtime contract.** The comment block above `build_middlewares()` isn't small talk — it's documenting dependencies the type system can't express: `ThreadData` must come before `Sandbox`, `Uploads` must come after `ThreadData`, `Clarification` must be registered last. Changing the order changes behavior, and nothing will warn you that you got it wrong.

**Second, registration order is not each hook's execution order.** One middleware can attach to several hooks; within a single hook, wrappers also have an outer/inner relationship. So this chapter doesn't read the list top to bottom — it reads by the run's lifecycle.

<MarginNote>Same code, different run, different chain. Plan mode, the vision model, tool_search, the subagent switch, the guardrail config — each changes which middleware actually get assembled.</MarginNote>

## How to read it: by run phase, not by file order

Middleware can attach to several kinds of hook. This part covers only the three that run before the model call:

| Hook | When | Times per run | Covered here |
| --- | --- | --- | --- |
| `before_agent` | entering the agent graph | once | ThreadData, Uploads, Sandbox, DynamicContext, Todo, LoopDetection |
| `before_model` | before each model call | per turn | Summarization, Todo, ViewImage |
| `wrap_model_call` | wrapping one model call | per turn | ToolOutputBudget, DanglingToolCall, LLMErrorHandling, LoopDetection, SkillActivation, Todo, DeferredToolFilter |

The other three take the lead only after the model returns:

```text
after_model
wrap_tool_call
after_agent
```

That's where the "onion" comes from: on the way in, middleware prepare state, messages, and the model request; after the model returns, the result travels back out, passing through policy, safety, tool execution, and side-effect handling in turn. Let's walk the inbound half first.

## before_agent: the run's opening

`before_agent` runs once, when the agent graph is entered. It handles run-level preparation: settle paths, surface uploaded files, set up the sandbox lifecycle, inject this turn's context, or clean up per-run state.

The full cast at this layer:

```text
ThreadData
Uploads
Sandbox
DynamicContext
Todo(before_agent cleanup, plan mode only)
LoopDetection(before_agent reset, if enabled)
```

**ThreadData** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/thread_data_middleware.py" sha="d2cc991d" symbol="ThreadDataMiddleware" />. From `thread_id` and an optional `user_id` it computes:

```text
workspace_path
uploads_path
outputs_path
```

To be precise: with the default `lazy_init=True`, it mostly **computes and writes the paths** — it doesn't necessarily create the directories right away. Only `lazy_init=False` eagerly creates them. In other words, `ThreadDataMiddleware` isn't a "filesystem tool" — it's more like a coordinate system for the middleware and tools that follow: where this thread's files should go.

It also stamps `run_id` and a timestamp onto the latest user message, so the frontend or runtime can identify this turn's input. That move quietly reveals an easily-missed fact: middleware can rewrite message objects right at the run's opening — it doesn't have to wait until just before the model call.

**Uploads** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/uploads_middleware.py" sha="d2cc991d" symbol="UploadsMiddleware" />. It is *not* responsible for "uploading files" — the frontend or gateway has already placed them in the thread's uploads directory. Its job is to scan the new files attached to the current message, plus the existing history files in that directory, build a `<uploaded_files>` context block, and prepend it before the latest `HumanMessage`.

This step is assembled only for the lead agent. Subagents don't install `UploadsMiddleware`, because a subagent inherits an execution context and doesn't need to re-process the user's upload entry point.

There's also a design pattern buried here that many later middleware reuse:

```text
the real file:
  stays in the thread uploads directory

what the model sees:
  filename, size, virtual path, outline/preview

what the frontend needs:
  structured info kept in message.additional_kwargs
```

So Uploads doesn't blindly stuff file contents into the prompt — it tells the model "which files exist, where they are, what type, roughly what they look like." If the model truly needs the full content, it reads it later through a file tool.

**Sandbox** comes from <SourceRef file="backend/packages/harness/deerflow/sandbox/middleware.py" sha="d2cc991d" symbol="SandboxMiddleware" />. This is the easiest one to get wrong: it too defaults to `lazy_init=True`. So `before_agent` usually does *not* acquire a sandbox on the spot; the real acquire may happen on the first sandbox-tool call.

A more accurate description of this layer: it establishes the sandbox's lifecycle and state contract.

```text
state["sandbox"] holds the sandbox_id
the sandbox provider manages the real sandbox instance
after_agent handles release
wrap_tool_call persists a sandbox_id created during tool execution back into graph state
```

Hold on to one division of labor:

```text
ThreadData → which thread a file path belongs to
Sandbox    → which execution environment a tool command runs in
```

The concrete isolation boundaries of local / container / Kubernetes / AIO sandboxes are for the later sandbox chapter. Here, anchor one key distinction:

```text
sandbox_id:
  the resource identity written into graph state

Sandbox instance:
  the real execution object the provider holds
```

Inside a tool, `runtime.state["sandbox"]` only gives you something like `{"sandbox_id": "..."}`; the thing that actually runs commands and reads/writes files is the sandbox instance the provider looks up from that id. This layering matters: state stores the identity, the provider owns the resource.

**DynamicContext** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py" sha="d2cc991d" />. It injects the date and memory as a hidden `HumanMessage` rather than editing the system prompt.

The reason for not touching the system prompt is concrete: the prefix cache. The moment the system prompt changes, the provider's prefix cache is void. Dynamic content like the date and memory has to ride the message stream. DynamicContext does it with an ID-swap, placing the reminder ahead of the user's question while keeping the user's original text recoverable and frontend-hideable.

It also has a reliability touch: the async path offloads the possibly-blocking memory build into `asyncio.to_thread()`, backstopped by a 5-second timeout. That way a slow memory build won't drag the whole run down.

The injection strategy isn't "drop one in every turn":

```text
first turn:
  inject date + memory before the first real user message

same day:
  inject nothing

date changed:
  add a date-only reminder before the latest real user message
```

There are two semantic layers to keep apart here:

```text
content envelope:
  <system-reminder>...</system-reminder>
  for the model to read — helps it see this isn't the user's ordinary text

additional_kwargs:
  markers like hide_from_ui / dynamic_context_reminder
  for the frontend and middleware to read — used to hide, identify, dedupe
```

So "model-visible" and "frontend-visible" are not the same thing. The model can see DynamicContext's reminder, but metadata tells the frontend not to render it as a user message.

<Design30>
Date and memory are not the same reliability class. The date is a cheap local fact; memory may involve storage, tokenization, and cold-start I/O. Today they share one timeout boundary; 3.0 could split them, so a memory timeout doesn't take date injection down with it.
</Design30>

**Todo** and **LoopDetection** only do state cleanup in `before_agent`: Todo clears completion reminders left over from other runs; LoopDetection initializes or resets this run's loop-detection state. They aren't the stars of this phase, but they're part of the full run skeleton.

## before_model: before each model call

`before_model` runs before every model call. It's the right place for two jobs:

```text
1. control the context budget
2. fold what the model should see this turn into the messages
```

The full cast at this layer:

```text
Summarization
Todo(before_model reminder, plan mode only)
ViewImage
```

**Summarization** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py" sha="d2cc991d" />. When history grows too long, it doesn't simply truncate — it replaces the messages wholesale:

```python
{
    "messages": [
        RemoveMessage(id=REMOVE_ALL_MESSAGES),
        *new_messages,       # one name="summary" summary
        *preserved_messages, # the recent window, kept
    ]
}
```

The summary itself is a `HumanMessage(name="summary")`. The `HumanMessage` role lets the model read it as background; `name="summary"` tells the runtime and middleware it isn't real user input — so later middleware won't treat the summary as a user prompt and parse slash commands or inject context against it.

On top of that, DeerFlow adds a structure-aware retention policy: if the to-summarize region holds a recent tool result that read `/mnt/skills`, the related AIMessage/ToolMessage bundle gets rescued from the to-summarize region back into the preserved region. The reason is that skill content is usually exact executable instruction, and flattening it into prose loses paths, constraints, and formatting.

That rescue isn't unlimited — it has a budget:

```text
preserve_recent_skill_count:
  at most how many recent skill bundles to rescue

preserve_recent_skill_tokens:
  total token budget for rescued content

preserve_recent_skill_tokens_per_skill:
  per-skill token cap
```

This keeps summarization from being defeated by rescuing "all the old skill content at once." The design judgment behind it: old messages aren't all disposable, and recent messages aren't always enough; what matters is whether this content still carries an active execution constraint.

<Design30>
Summarization is not "generic compression" — it's "selective loss." Ordinary chat can be summarized; still-active instructions, schemas, tool results, and constraints are closer to executable context. 3.0 could first extract typed context items, then decide per item whether to keep raw, summarize, externalize, or move to long-term memory by retention value.
</Design30>

**Todo** in `before_model` fills a different kind of context: if the todo state still exists but the original `write_todos` call has been pushed out of the current messages by summarization or window trimming, it injects a hidden reminder telling the model there's still an active todo list.

The essence here: `state["todos"]` is still alive, but the message history the model can see no longer shows where the todos came from. The runtime knows the task isn't done, yet the model might think it can wrap up. What TodoMiddleware does is turn task state that "the runtime still remembers but the message window has lost" back into a reminder the model can see.

Note the similarity and difference with DynamicContext:

```text
DynamicContext:
  fills external context like the current date / memory

Todo before_model:
  fills task context that's still alive in graph state but lost from the message window
```

**ViewImage** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/view_image_middleware.py" sha="d2cc991d" />, and is assembled only when the model supports vision. It turns images the model has already read through the `view_image` tool into a provider-compatible multimodal message, so the next model call can actually "see" the image content rather than just a "tool succeeded" line.

Keep this separate from the tool itself:

```text
view_image tool:
  reads the image, writes image info into state["viewed_images"]
  returns ToolMessage("Successfully read image")

ViewImageMiddleware:
  reads state["viewed_images"] in the before_model phase
  builds a provider-compatible multimodal HumanMessage
```

So a finished tool call only means the runtime got the image data; whether the model can "see" those images next turn still depends on middleware injecting them into the model request.

<Design30>
ViewImage exposes a general problem: don't equate "the last HumanMessage" with "the last user input." The message list can be laced with runtime injections — dynamic context, todo reminders, summaries, image projections. 3.0 needs clearer provenance for runtime-injected messages (source, purpose, idempotency key), instead of leaning on prompt wording to carry control metadata.
</Design30>

## wrap_model_call: wrapping the model call

`wrap_model_call` is the innermost layer, closest to the model provider. It receives a `handler`, and `handler(request)` means "keep going inward and eventually call the real model." You can change the request before calling `handler`; you can process the response after `handler` returns; and if `handler` raises, you enter the exception path.

This layer involves more middleware than it looks:

```text
ToolOutputBudget(model side)
DanglingToolCall
LLMErrorHandling
LoopDetection(wrap_model_call soft warning)
SkillActivation
Todo(wrap_model_call completion reminder)
DeferredToolFilter(model side)
```

**ToolOutputBudget** does a last-resort trim on the model side. It checks the history for oversized `ToolMessage`s and swaps their content for a head/tail preview plus an external path, so the next model call isn't dragged down by a huge tool result. It's not a summarizer — the full output is still kept in a file where possible.

This is easy to confuse with Summarization:

```text
Summarization:
  conversation history boundary
  replaces old conversation with a summary

ToolOutputBudget:
  single tool result boundary
  externalizes the full raw output, leaves only a preview in the ToolMessage
```

The head/tail preview isn't a semantic summary either — it's a navigation signal:

```text
head:
  helps the model judge what this output is and how it's structured

tail:
  helps the model see the ending status, errors, summary line

path:
  if the full content is really needed, read it with read_file
```

So it protects the model's context budget, not the model's understanding of the file.

**DanglingToolCall** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/dangling_tool_call_middleware.py" sha="d2cc991d" />, and repairs the provider's tool-call protocol. It guards against a history like:

```text
AIMessage(tool_calls=[call_1])
  ...missing ToolMessage(tool_call_id=call_1)
```

Many providers require every `AIMessage.tool_calls[*].id` to be followed by a matching `ToolMessage.tool_call_id`. DanglingToolCall's algorithm is "index first, rebuild second":

```text
tool_messages_by_id:
  dict[tool_call_id, deque[ToolMessage]]

tool_call_ids:
  every tool_call id any AIMessage declared

patched:
  the rebuilt request.messages for this call
```

The first step groups existing ToolMessages by their own `tool_call_id`:

```python
tool_messages_by_id[msg.tool_call_id].append(msg)
```

The second step, on reaching an AIMessage, looks up that index in the order of its `tool_calls`. A real ToolMessage found gets moved right after the AIMessage; nothing found means a synthetic `ToolMessage(status="error")` is appended. So the final order isn't guaranteed by the physical position of ToolMessages in the original history — it's rebuilt from the protocol relationship `AIMessage.tool_calls[].id == ToolMessage.tool_call_id`.

One more detail worth noticing: when `_message_tool_calls()` collects tool-call ids, it checks three sources at once — `msg.tool_calls`, `msg.additional_kwargs["tool_calls"]`, and `msg.invalid_tool_calls`. Different providers' adapters may stash tool-call info in different fields, and some even mark a parse-failed call as `invalid_tool_calls`. DanglingToolCall's principle is to over-repair rather than under-repair: as long as some AIMessage claims it emitted a tool call — whichever field it lives in — there must be a matching ToolMessage after it.

Note it patches *this* `ModelRequest`, not graph state directly:

```text
request.override(messages=patched)
```

That's exactly why it chooses `wrap_model_call` over `before_model`: the patch has to be inserted right after the corresponding AIMessage, not simply appended to the end.

**LLMErrorHandling** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/llm_error_handling_middleware.py" sha="d2cc991d" />. It guards the model-provider boundary: timeouts, connection failures, 5xx, 429, quota/auth — all classified and handled here. Retriable transient/busy errors back off and retry; non-retriable or retry-exhausted ones return a fallback `AIMessage`.

It handles exceptions raised by `handler(request)` — the "didn't get a normal AIMessage" case. It does *not* handle the case where the provider returned normally but the content is untrustworthy. That boundary matters: the next stop's `SafetyFinishReasonMiddleware` is what handles "the provider returned an AIMessage normally, but finish_reason says it was safety-terminated, so its tool_calls can't be trusted."

It also has a circuit breaker:

```text
closed     circuit connected, model calls pass through
open       circuit open, fast-fail to fallback
half_open  recovery probe, lets one call test whether the provider recovered
```

Note that `closed` here is the healthy state, not "requests closed off" — that's the circuit-breaker industry term, not DeerFlow writing it backwards. Another key point: `GraphBubbleUp` must be re-raised as-is, because it represents LangGraph's interrupt/pause/resume control flow, not an ordinary exception — it can't be swallowed as a provider error.

**LoopDetection** in `wrap_model_call` mainly injects a soft warning: if the previous `after_model` judged a loop risk but it hasn't reached a hard stop, it slips a reminder into the next model request. It doesn't inject straight from `after_model` because that would break the AIMessage(tool_calls) → ToolMessage pairing — the warning has to wait until the tool results are all back.

**SkillActivation** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/skill_activation_middleware.py" sha="d2cc991d" />. When the user starts with `/skill-name`, it reads the full `SKILL.md` and injects a hidden `HumanMessage` into this call's request. That content doesn't persist into graph state — it's a turn-local request overlay. This way a skill enters the context only when explicitly activated, instead of polluting the prompt every turn.

Its difference from DynamicContext is worth pulling out:

```text
DynamicContext:
  before_agent
  writes graph state messages
  conversation-level context

SkillActivation:
  wrap_model_call
  only overrides the current ModelRequest.messages
  turn-local context
```

SkillActivation also wraps the `SKILL.md` in an XML-like envelope:

```text
<slash_skill_activation>
  <user_request>...</user_request>
  <skill ...>
    <skill_content encoding="xml-escaped">...</skill_content>
  </skill>
</slash_skill_activation>
```

Here `xml-escaped` means the skill content's `<`, `>`, `&` and friends are escaped into plain text (the implementation uses Python's `html.escape()`), so tag-like content inside the skill document can't break the outer prompt envelope.

If the skill doesn't exist, is disabled, or the current agent isn't allowed to use it, the middleware returns an `AIMessage` it builds itself — without spending a model request.

**Todo** in `wrap_model_call` consumes the completion reminders queued during the `after_model` phase. If the model tries to wrap up while todos are incomplete, Todo's `after_model` jumps back to the model first; the next `wrap_model_call` then injects the hidden reminder into that turn's request.

**DeferredToolFilter** comes from <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/deferred_tool_filter_middleware.py" sha="d2cc991d" />. Its enforcement is two-layered, and you need both:

```text
wrap_model_call (model side):
  filter hidden schemas out of request.tools
  the model can't see these tools, so it won't call them on its own

wrap_tool_call (tool side):
  if the model calls a hidden tool anyway
  (guessing the name from history or a prompt leak, say)
  return ToolMessage(status="error") and don't execute
```

Filtering on the model side alone isn't enough — the model can still guess an unpromoted tool's name from history, a prompt hint, or provider quirks. It's only enforcement when both layers hold.

<Design30>
The point of DeferredToolFilter isn't just "hide the schema" — it's two-layer enforcement: the model side controls visible schemas, the tool side controls what can actually execute. The first alone isn't enough — the model can still call an unpromoted tool from history or guessing. 3.0 could abstract this into a first-class `ToolVisibilityPolicy` that declares visible, executable, and blocked tools at once.
</Design30>

<Philosophy title="Design rationale">

The inbound half isn't reasoning — it's preparing the model call: runtime state in place, model-visible context in place, message protocol in place, and the request's tool schemas and large outputs tidied up too.

DeerFlow doesn't push this cross-cutting logic into the graph nodes; it puts it in middleware. The upside is that the graph stays a clean "model ↔ tools" main loop, easy to read; the cost is that order becomes an implicit interface — put one middleware on the wrong hook, or in the wrong registration order, and the code still runs while the semantics have already changed.

</Philosophy>

<Tradeoff leftTitle="What inbound layering buys" rightTitle="The cost">
  <Fragment slot="left">

  - cross-cutting prep stays out of the graph; the main loop stays simple
  - lead and subagent can reuse the runtime base
  - dynamic content rides the message stream; the system prompt stays stable, prefix-cache friendly

  </Fragment>
  <Fragment slot="right">

  - order is an implicit convention, held mostly by comments and tests
  - the chain varies by run; debugging starts with reconstructing this run's assembly
  - a sync hook doing blocking I/O stalls the shared event loop

  </Fragment>
</Tradeoff>

## Easy traps

- **`before_agent` runs once; `before_model` runs every turn.** Paths, uploads, and the sandbox lifecycle are run-level prep; summaries, todo reminders, and image injection ride each model call.
- **Default lazy init fools your intuition.** `ThreadData` only computes paths by default, and `Sandbox` doesn't acquire on the spot in before_agent. Don't read "middleware registered" as "resource already created."
- **The chain is assembled conditionally.** Plan mode, the vision model, tool_search, the subagent switch, the guardrail provider — all change the actual middleware list. Before debugging, confirm: is this middleware even in *this* run?
- **`wrap_model_call` ≠ `before_model`.** DanglingToolCall must insert its patch at the right position in history, so it can only edit this request; `before_model` is better suited to returning a graph-state update.
- **The system prompt is kept stable on purpose.** Date, memory, skill, todo, images — this dynamic info all rides the message / request overlay. Don't casually edit the system prompt.

---

This stop only goes as far as the model call. The request has crossed `before_agent`, `before_model`, and `wrap_model_call`, and the model finally holds a tidied request. The next stop walks the other half: after the model returns, `after_model` takes the result in reverse order, tool execution passes through `wrap_tool_call`, and the run ends in `after_agent`. Safety policy, loop detection, the tool boundary, and side effects — the dense half happens over there.
