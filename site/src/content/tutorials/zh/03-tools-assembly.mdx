第一次读 tools 模块，很多人会以为这里只是注册了一批函数。这个理解只说对了一部分。关键问题是：为什么有的工具明明配置了，却没有暴露给模型？顺着这个问题往下看，才会发现工具的「可见性」不是由单个列表决定的。

先抓住一个判断，后面就容易对齐：**工具注册了，不代表这次 agent 就能用。** 配置、内置能力、模型能力、沙箱、skill、MCP 和子 agent 策略共同决定这次 run 能动用哪些工具、哪些工具会被过滤、哪些 schema 暂时不暴露给模型。完成这次计算后，才得到传给 `create_agent(..., tools=...)` 的最终列表。

## 候选集，不是最终集

主入口在 <SourceRef file="backend/packages/harness/deerflow/tools/tools.py" sha="7e7f0410" /> 的 `get_available_tools()`。但请注意：**它产出的还不是最终工具列表，只是候选集。** 它回答的是一个有限的问题：

```text
按系统配置、内置能力和 runtime 开关，这次 agent 构建最多能拿到哪些候选工具？
```

它大致按这个顺序走：从 `AppConfig.tools` 起步 → 按自定义 agent 的 `tool_groups` 过滤 → 不允许 host bash 就先过滤掉 → 用 `resolve_variable(cfg.use, BaseTool)` 动态导入 → 补内置工具 → 按开关加 `task_tool`、按模型能力加 `view_image_tool` → 接入 MCP / ACP → 最后**按 `tool.name` 去重**。

之后回到 <SourceRef file="backend/packages/harness/deerflow/agents/lead_agent/agent.py" sha="7e7f0410" /> 的 `_make_lead_agent()`，候选集还要再过两道：skill 级的 `allowed_tools` 过滤，和 MCP deferred tool 的装配。所以「最终暴露什么」是**两段式**的——先收集候选，再应用策略。

## 配置不是工具本身

`config.yaml` 里的工具长这样：

```yaml
tools:
  - name: read_file
    group: file:read
    use: deerflow.sandbox.tools:read_file_tool
```

这里的 `use` 是 DeerFlow 的约定，格式是 `module.path:variable_name`，不是 Python 语法。`resolve_variable()` 把冒号前交给 `import_module()`，再从模块里取冒号后的变量，并校验它确实是个 `BaseTool` 实例。

这带来一个**关键区别**，记住它能少走很多排查弯路：

```text
配置里的 name  =  人读配置时的标签
tool.name      =  暴露给模型、用来路由的名字
```

最终路由看的是 `BaseTool.name`，不一定等于 `config.yaml` 里写的 `name`。两者一旦不一致，模型看到的 schema 名和运行时路由名就会错位——DeerFlow 会为此打 warning。

<MarginNote>动态导入很灵活，代价是失败得晚：模块路径写错、变量名写错、可选依赖没装，或者取出来的根本不是 BaseTool，都要等运行期解析时才报错。</MarginNote>

## 可见性是好几道门，不是一张白名单

「工具为什么不见了」之所以难查，是因为它要经过好几层策略，每一层都在不同文件里：

- **`tool_groups`** —— 自定义 agent 的粗粒度筛选条件。配了就只留对应 group 的工具。
- **host bash 安全闸** —— 即使 `config.yaml` 配了 `bash`，也要先过 `is_host_bash_allowed(config)`；不允许就过滤掉。这是条**安全边界**，不是普通开关。
- **模型能力** —— `view_image_tool` 只有在选中的模型声明 `supports_vision` 时才追加。
- **skill 策略** —— <SourceRef file="backend/packages/harness/deerflow/skills/tool_policy.py" sha="7e7f0410" /> 的 `filter_tools_by_skill_allowed_tools()`：没有任何 skill 声明 `allowed_tools` 时保持「全放行」，一旦有 skill 显式声明，就只保留声明集合里的名字。这是最小权限在工具层的落点。

## 工具不止一种形态

读 tools 模块的第二个误区，是以为每个工具都只是「函数调用」。DeerFlow 的工具至少有四种形态，先认清它们，后面几章会更容易理解：

- **普通执行工具** —— `@tool` 包出来的 `BaseTool`，模型发 `tool_call`、ToolNode 执行函数。`read_file`、`web_search`、`bash` 都是。
- **状态更新工具** —— `present_files` 返回 LangGraph `Command(update={"artifacts": ...})`，**直接写入图状态**；前端再据此渲染产物。
- **控制流工具** —— `ask_clarification` 的函数体只是个占位，行为由 `ClarificationMiddleware` 接管：它拦下调用、`Command(..., goto=END)` 停住当前 run，等用户下一条消息。**需要人补充信息时，应当把它建模成一次「中断」，不要让工具阻塞等待。**
- **委派工具** —— `task` 是子 agent 入口：继承父级沙箱/模型/工具策略，用 `subagent_enabled=False` 构造子工具集（避免递归暴露 `task`），后台跑 `SubagentExecutor`，再把结果当 tool result 还回来。

<MarginNote>这里有一个设计取舍：父 agent「能不能委派」，直接表现为工具列表里有没有 task_tool。能力开关和工具列表修改耦在一起——子 agent 那一章会专门拆这件事。</MarginNote>

## 工具存在，schema 却不一定暴露

MCP 工具最能说明「在列表里，也不一定会暴露给模型」这件事。打开 `tool_search.enabled` 后，MCP 工具**不会**把完整 schema 一次性塞给模型——prompt 里只列名字，等模型需要时再调 `tool_search` 取 schema。理由很实际：

```text
一个 MCP server 可能暴露几十个工具。
全量绑定 schema 会让 prompt 变大、成本变高，也会增加模型选择工具时的噪声。
```

这条链路横跨多个模块，正是 DeerFlow「一个能力由多层协作完成」的典型（<SourceRef file="backend/packages/harness/deerflow/tools/builtins/tool_search.py" sha="7e7f0410" />）：

```text
get_available_tools  ->  tag_mcp_tool(t) 打标
assemble_deferred_tools  ->  建 catalog + tool_search，返回 DeferredToolSetup
apply_prompt_template  ->  prompt 只列 deferred 工具名
model 调 tool_search  ->  返回匹配 schema + 写 state.promoted
DeferredToolFilterMiddleware  ->  未 promoted 的 schema 不给模型；直接调就回 error
```

`ThreadState.promoted` 的 reducer 用 `catalog_hash` 做作用域：hash 没变就合并 promoted 名字，hash 变了就替换——这样旧状态里的同名 promotion，不会在 MCP 配置变化后错误地放开另一个工具。

MCP 还有一层运行时适配（<SourceRef file="backend/packages/harness/deerflow/mcp/tools.py" sha="7e7f0410" symbol="_make_session_pool_tool" />）：stdio MCP server 会按 `user_id + thread_id` 复用 session，并把进程 `cwd` 和临时目录固定在线程 workspace 下。这样 Playwright 这类 MCP 工具生成截图、trace 或临时文件时，文件会落在 `/mnt/user-data/...` 能映射到的位置；工具结果里出现的本地文件路径，也会尽量改写成虚拟路径。也就是说，MCP 除了延迟暴露 schema，还要把外部工具产生的文件重新接回 DeerFlow 的 sandbox 路径体系。

<Philosophy title="工程设计哲学">

整章最该带走的，是这个**两段式**的刻意分工：

`get_available_tools()` 回答「这次 run 有哪些候选工具」；`filter_tools_by_skill_allowed_tools()` 回答「skill 加载后，还允许哪些」。把「工具来源」和「工具策略」分开，可见性才能**同时**被 agent 级配置和 skill 级最小权限两套规则控制，而不互相覆盖。

再往上抽一层：**工具的可见性来自一次策略计算，不能按静态注册表理解。** 以后排查「工具不见了」，不要只问「函数在不在」；更关键的是确认「在哪一层、被哪条策略、按什么理由过滤掉了」。这也是 3.0 想把 `ToolRegistry / ToolPolicy / ToolRuntime` 拆开的动机：注册、策略、执行，本就是三个问题。

</Philosophy>

## 易误解的要点

- **`config name` 不等于 `tool.name`。** 最终路由看 `BaseTool.name`。两者不一致时，模型看到的名字和运行时路由的名字会错位。
- **host bash 是安全边界，不是普通开关。** 配了不代表放行，还必须通过 `is_host_bash_allowed(config)`。
- **MCP 不只涉及 schema。** deferred 解决“模型先别看到所有 schema”；stdio MCP 的 session、cwd、temp dir、输出路径改写解决“外部工具生成的文件如何回到 thread workspace”。
- **「可见」不等于「可执行」。** deferred MCP 工具仍被 ToolNode 持有、可执行，只是 schema 先不暴露给模型——没 promoted 就直接调，会被中间件挡回。
- **去重有优先级。** 顺序是 config → 内置 → MCP → ACP，同名时**先出现的保留**。因为路由按名字走，重名会让 schema 歧义。

---

工具装配这一章回答的是：这次 run 的工具候选从哪里来，哪些 schema 会暴露给模型，哪些调用会被策略挡住。它不展开某个工具函数的执行细节。把来源、可见性和执行策略分开，整个工具系统就容易读多了。等工具齐了、图也装好了，下一个会影响运行行为的变量是：**中间件的顺序**。下一站，我们就看这个顺序怎么变成运行时语义。
