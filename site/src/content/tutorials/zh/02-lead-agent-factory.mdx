上一章里，用户输入已经被 Gateway 登记成一次 run。现在它进入 `make_lead_agent()`：这一章看 DeerFlow 如何把「运行时选项 + 应用配置」装配成一张可运行的 LangGraph agent 图。

先记住一件事，后面所有细节都围绕它展开：**这里负责把运行时选项和应用配置装配成 agent 图。** 推理发生在模型、工具和中间件的交互里；这个函数只负责把零件按规矩装好，然后把图交出去，自己不执行图。

## 一个函数，两个名字

打开 <SourceRef file="backend/packages/harness/deerflow/agents/lead_agent/agent.py" sha="7e7f0410" />，你会看到一对名字很接近的函数：`make_lead_agent()` 和 `_make_lead_agent()`。那个下划线不是装饰，它划出了整篇最重要的一条边界。

**`make_lead_agent(config)`** 只收一个参数，因为 LangGraph Server 要求图工厂必须能「只凭一个 config」被调起（见 <SourceRef file="backend/langgraph.json" sha="7e7f0410" />）。它的职责很薄：解析出 `AppConfig`（优先使用运行时注入的配置，回退到 `get_app_config()`），然后把装配工作交给下划线版本。它是**兼容适配器**。

**`_make_lead_agent(config, *, app_config)`** 是实际的装配函数。那个 `*` 把 `app_config` 设成只能按关键字传——因为两个参数语义上都像 config，位置传参太容易搞混。它做的是确定性的事：从运行时选项 + 应用配置，一步步把图组装出来。

<MarginNote>下划线在 Python 里没有任何强制力，只是一种约定。但这里它确实有讲究：外层函数处理兼容，内层函数负责装配。</MarginNote>

## 运行选项是怎么拼出来的

`_get_runtime_config()` 把 `config["configurable"]` 和 `config["context"]` 合并。注意这里有明确的覆盖顺序：先铺 `configurable`，再让 `context` 覆盖同名键。

它读出的核心字段是**每次 run 的选项**，不是全局应用设置：

```text
model_name / model
thinking_enabled
reasoning_effort
is_plan_mode
subagent_enabled
max_concurrent_subagents
is_bootstrap
agent_name
```

## 模型怎么选

模型解析有明确的优先级：

```text
请求里的 model_name
  > 自定义 agent 配置里的模型
  > AppConfig.models[0]
```

如果 `thinking_enabled=True`，但选中的模型并不支持 thinking，工厂会记一条 warning，然后把 `thinking_enabled` 关掉。想开启 thinking mode，也得看模型支不支持。

## 最终配方

所有零件就位后，最终调用会收束成这样：

```python
create_agent(
    model=create_chat_model(...),
    tools=filtered_tools,
    middleware=build_middlewares(...),
    system_prompt=apply_prompt_template(...),
    state_schema=ThreadState,
)
```

心智模型一句话：

```text
Lead Agent 图 = 模型 + 工具 + 中间件 + system_prompt + ThreadState
```

`create_agent()` 提供标准的「模型 ↔ 工具」循环；DeerFlow 在这里负责备齐**材料**（模型、工具）和**运行约束**（中间件、prompt、状态结构）。

<Philosophy>

为什么 `make_lead_agent` 要多这一层？因为它同时要兼容好几个宿主：官方 LangGraph Server / Studio、DeerFlow Gateway、嵌入式客户端、测试，还要兼容「老的 `configurable`」和「新的 `context`」两套约定。

`app_config` 既能从运行时 config 里取，又能全局回退。这条绕路，是为了在**不破坏 LangGraph Server ABI** 的前提下，让 Gateway 注入一份热加载的配置快照。

设计要点是：**兼容边界要显式、要单点。** 把所有兼容逻辑集中到一个命名清楚的适配函数里，让主体装配逻辑保持确定、清晰。这比让兼容逻辑散落在多处更容易维护。

</Philosophy>

## 易误解的要点

- **`cfg` 是个重载名。** 它装着每次 run 的选项，也可能混进 `app_config`、`thread_id`、`run_id` 这类基础设施字段。看到 `cfg` 时，不要只按变量名判断它的语义。
- **`config` 会被就地修改。** 工厂会写入 `metadata` 和 `callbacks`，不要把它当只读对象。
- **bootstrap 和自定义 agent 容易混。** `setup_agent` 只在 bootstrap 流程出现；`update_agent` 只在已有自定义 agent 的对话里。如果混淆这两个场景，就会暴露错误的工具。
- **中间件顺序在别处装配，却是最终行为的一部分。** 改这条装配线，可能无意中改了运行语义——这会在中间件章节展开。

---

装配线的价值在于**确定性**：它按配置组装图，不在这里做推理，也不临时改变策略。下一章先看工具列表如何被筛出来；因为模型能调用什么，决定了这张图实际能做什么。
