第一次打开 `build_middlewares()`，它看起来只是把一组对象 append 到 list 里。但这个 list 实际决定了一次 run 的运行时语义：模型不是直接拿到用户 prompt，而是先经过一组 middleware。它们会准备 thread 文件路径、处理上传文件、建立 sandbox 生命周期契约、注入日期和记忆、修补 provider 要求的消息协议，并在真正调用模型前调整本轮 request。

上一站讲的是工具如何被收集和暴露。这一站继续往里走：**一次 run 如何穿过 middleware 洋葱，抵达 model call。** 模型返回之后的另一半，包括 `after_model`、`wrap_tool_call`、`after_agent`，放到下一站。

<MentalModel>模型调用发生前，请求已经被 middleware 整理成 DeerFlow 希望模型看到的形状。</MentalModel>

## 一个 list，两段拼出来

装配入口是 <SourceRef file="backend/packages/harness/deerflow/agents/lead_agent/agent.py" sha="4a2f0a23" symbol="build_middlewares" />。它先取一段 lead 和 subagent 复用的 runtime 基座，再叠上 lead agent 专属的 middleware。

共享基座来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/tool_error_handling_middleware.py" sha="4a2f0a23" symbol="build_lead_runtime_middlewares" />：

```text
共享基座（lead 与 subagent 复用）
  ToolOutputBudget    大工具输出落盘，消息里只留预览和路径
  ThreadData          计算 thread_data 路径，给 workspace/uploads/outputs 定位
  Uploads             把上传文件信息投影给模型（只有 lead 有）
  Sandbox             管理 sandbox 生命周期和 sandbox_id 持久化
  DanglingToolCall    修补历史里悬空的工具调用协议
  LLMErrorHandling    模型调用失败时重试、熔断、兜底
  Guardrail?          可选运行时授权（配置了 provider 才有）
  SandboxAudit        bash command 的 pass/warn/block 闸门
  ToolErrorHandling   工具异常转成 ToolMessage(status="error")
```

然后 `build_middlewares()` 继续追加 lead 专属段：

```text
lead 专属段（带 ? 的按条件装配）
  DynamicContext      注入日期和记忆，保持 system prompt 不变
  SkillActivation     用户用 /skill 时，把完整 SKILL.md 注入本轮 request
  Summarization?      历史太长时压缩旧消息
  TodoList?           plan mode 的任务清单和完成提醒
  TokenUsage?         记录 token usage，并给动作做产品语义归因
  Title               第一轮完整对话后生成标题
  Memory              run 结束后把对话送入 memory queue
  ViewImage?          视觉模型可用时，把已查看图片投影给模型
  DeferredToolFilter? 隐藏未 promoted 的 deferred tool schema，并拦截非法调用
  SubagentLimit?      限制并行 task/subagent 数量
  LoopDetection?      检测并打断重复工具调用循环
  (custom middleware)
  SafetyFinishReason? provider safety stop 后清空不可信 tool_calls
  Clarification       把 ask_clarification 变成 run 级暂停，始终最后注册
```

这里要抓住两点。

**第一，顺序是运行时契约。** `build_middlewares()` 上方的注释不是普通说明，它在表达源码无法用类型系统描述的依赖：`ThreadData` 要在 `Sandbox` 前面，`Uploads` 要在 `ThreadData` 后面，`Clarification` 要最后注册。改顺序就是改行为。

**第二，注册顺序不等于每个 hook 的执行顺序。** 同一个 middleware 可能挂多个 hook；同一个 hook 内部还可能有 wrapper 的外内关系。所以这章不按 list 从上到下读，而是按 run 生命周期读。

<MarginNote>同一份代码，不同 run 的链可能不同。plan mode、vision model、tool_search、subagent 开关、guardrail 配置都会改变实际装配出来的 middleware。</MarginNote>

## 读法：按 run phase，不按文件顺序

middleware 一共有几类 hook。这一篇只讲模型调用前的三类：

| Hook | 时机 | 一次 run 跑几次 | 本章涉及 |
| --- | --- | --- | --- |
| `before_agent` | 进入 agent graph 时 | 1 次 | ThreadData、Uploads、Sandbox、DynamicContext、Todo、LoopDetection |
| `before_model` | 每轮 model call 前 | 每轮 | Summarization、Todo、ViewImage |
| `wrap_model_call` | 包住一次 model call | 每轮 | ToolOutputBudget、DanglingToolCall、LLMErrorHandling、LoopDetection、SkillActivation、Todo、DeferredToolFilter |

剩下三类在模型返回后才成为主角：

```text
after_model
wrap_tool_call
after_agent
```

这就是“洋葱”模型的意义：请求向内走时，middleware 准备 state、messages 和 model request；模型返回后，回答再向外走，经过策略、安全、工具执行和副作用处理。下面先走向内半程。

## before_agent：run 开场

`before_agent` 在进入 agent graph 时跑一次。它适合做 run 级准备：确定路径、投影上传文件、建立 sandbox 生命周期、注入本轮开场上下文，或者清理 per-run 状态。

这一拍的全量清单是：

```text
ThreadData
Uploads
Sandbox
DynamicContext
Todo(before_agent cleanup, plan mode only)
LoopDetection(before_agent reset, if enabled)
```

**ThreadData** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/thread_data_middleware.py" sha="4a2f0a23" symbol="ThreadDataMiddleware" />。它根据 `thread_id` 和可选的 `user_id` 计算：

```text
workspace_path
uploads_path
outputs_path
```

这里要说准确：默认 `lazy_init=True` 时，它主要是**计算并写入路径**，不一定立刻创建目录。只有 `lazy_init=False` 时才会 eager create。也就是说，`ThreadDataMiddleware` 不是“文件系统工具”，它更像是给后续 middleware 和工具提供 thread file space（线程文件空间）的坐标。

它还会给最后一条用户消息补上 `run_id` 和 timestamp，方便后续前端或 runtime 识别这一轮输入。这个动作也说明一个容易忽略的事实：middleware 可以在 run 开场就改写消息对象，不一定等到 model call 前才动 messages。

**Uploads** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/uploads_middleware.py" sha="4a2f0a23" symbol="UploadsMiddleware" />。它不负责“上传文件”这个动作；文件已经由前端或 gateway 放进 thread 的 uploads 区。它做的是把当前消息里的新文件和目录里已有的历史文件扫描出来，生成 `<uploaded_files>` 上下文，并 prepend 到最后一条 `HumanMessage` 前面。

这一步只有 lead agent 装配。subagent 不装 `UploadsMiddleware`，因为 subagent 继承的是执行上下文，不重新处理用户上传入口。

这里还有一个和后面很多 middleware 共享的设计模式：

```text
真实文件:
  留在 thread uploads 目录

模型看到的内容:
  文件名、大小、虚拟路径、outline/preview

前端需要的结构化信息:
  保留在 message.additional_kwargs
```

也就是说，Uploads 不是把文件内容无脑塞进 prompt，而是把“文件存在、在哪里、是什么类型、有什么轮廓”投影给模型。模型如果真的需要完整内容，后续再通过文件工具读取。

**Sandbox** 来自 <SourceRef file="backend/packages/harness/deerflow/sandbox/middleware.py" sha="4a2f0a23" symbol="SandboxMiddleware" />。这里最容易说错：它默认也是 `lazy_init=True`。也就是说，`before_agent` 阶段通常不会马上 acquire 一个 sandbox；真正的 acquire 可能发生在第一次 sandbox tool 调用时。

这一层更准确的职责是建立 sandbox 的生命周期和状态契约：

```text
state["sandbox"] 里保存 sandbox_id
sandbox provider 管理真正的 sandbox 实例
after_agent 负责 release
wrap_tool_call 负责把工具执行中新产生的 sandbox_id 持久回 graph state
```

先记一个分工：

```text
ThreadData -> 文件路径属于哪个 thread
Sandbox    -> 工具命令在哪个执行环境里跑
```

具体 local/container/Kubernetes/AIO sandbox 的隔离边界，放到后面的 sandbox 章节再展开。

这里先保留一个重要区分：

```text
sandbox_id:
  写进 graph state 的资源身份

Sandbox instance:
  provider 持有的真实执行对象
```

工具里通过 `runtime.state["sandbox"]` 只能拿到类似 `{"sandbox_id": "..."}` 的状态；真正能执行命令、读写文件的是 provider 根据这个 id 找回来的 sandbox 实例。这个分层很关键：state 里存身份，provider 管资源。

**DynamicContext** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py" sha="4a2f0a23" />。它把日期和记忆注入为隐藏 `HumanMessage`，而不是改 system prompt。

原因是 prefix cache：system prompt 一旦变化，供应商的前缀缓存就无法复用。日期和 memory 这类动态内容必须走 message stream。DynamicContext 用 ID-swap（ID 互换）把 reminder 放到用户问题前面，同时保持用户原文可恢复、前端可隐藏。

它还有一个可靠性设计：异步路径会把可能阻塞的 memory 构建放进 `asyncio.to_thread()`，再用 5 秒超时限制。这样 memory 慢了不会拖死整个 run。

它的注入策略不是“每轮都塞一条”：

```text
first turn:
  在第一条真实用户消息前注入 date + memory

same day:
  不重复注入

date changed:
  在最新真实用户消息前补一条 date-only reminder
```

这里有两个语义层要分清：

```text
content envelope:
  <system-reminder>...</system-reminder>
  给模型读，帮助模型理解这不是用户普通正文

additional_kwargs:
  hide_from_ui / dynamic_context_reminder 等标记
  给前端和 middleware 读，用来隐藏、识别、去重
```

所以“模型可见”和“前端可见”不是一回事。DynamicContext 的 reminder 对模型可见，但通过 metadata 告诉前端不要当成用户消息展示。

<Design30>
日期和记忆不是同一种可靠性等级。日期是廉价本地事实，memory 可能涉及存储、分词和冷启动 IO。现在它们共用一个超时边界；3.0 可以拆开，让 memory 超时不影响日期注入。
</Design30>

**Todo** 和 **LoopDetection** 在 `before_agent` 里只是清理状态。Todo 会清掉其他 run 遗留的 completion reminder；LoopDetection 会初始化或重置本 run 的循环检测状态。它们不是这一拍的主角，但属于完整 run 骨架的一部分。

## before_model：每轮模型调用前

`before_model` 每次 model call 前都会跑。它适合做两类事：

```text
1. 控制上下文预算
2. 把模型本轮应该看到的内容投影进 messages
```

这一拍的全量清单是：

```text
Summarization
Todo(before_model reminder, plan mode only)
ViewImage
```

**Summarization** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py" sha="4a2f0a23" />。历史太长时，它不是简单截断，而是替换整段 messages：

```python
{
    "messages": [
        RemoveMessage(id=REMOVE_ALL_MESSAGES),
        *new_messages,       # 一条 name="summary" 的摘要
        *preserved_messages, # 保留的近段
    ]
}
```

摘要本身是 `HumanMessage(name="summary")`。`HumanMessage` 让模型把它当背景读，`name="summary"` 让 runtime 和 middleware 知道它不是用户真实输入。

DeerFlow 还做了一个结构化保留策略：如果待压缩区里有最近读取 `/mnt/skills` 的 tool result，会把相关 AIMessage/ToolMessage bundle 从待压缩区救回 preserved 区。原因是 skill 内容通常是精确执行指令，压成散文会丢路径、约束和格式。

这一步不是无限保留，它有预算：

```text
preserve_recent_skill_count:
  最多救回几个最近 skill bundle

preserve_recent_skill_tokens:
  救回内容的总 token budget

preserve_recent_skill_tokens_per_skill:
  单个 skill 的 token cap
```

这能防止 summarization 被“大量旧 skill 内容全部救回”击穿。设计上，它表达的是一个更通用的判断：旧消息不一定都该丢，近消息也不一定都够用；真正要看这段内容是不是还在承担执行约束。

<Design30>
summarization 不是“通用压缩”，而是“选择性丢弃”。普通聊天可以摘要；仍在生效的指令、schema、工具结果和约束更接近执行上下文。3.0 可以先抽取 typed context items，再按保留价值决定原文保留、摘要、外置或长期记忆。
</Design30>

**Todo** 在 `before_model` 里负责另一类补上下文：如果 todo state 还在，但原来的 `write_todos` 调用已经被 summarization 或窗口裁剪移出当前消息，它会注入隐藏 reminder，提醒模型当前还有 active todo list。

这里的本质是：`state["todos"]` 还在，但模型可见的消息历史里已经看不到 todo 是怎么来的。runtime 知道任务没做完，模型却可能以为自己可以收尾。TodoMiddleware 就把 durable-ish planning state（运行时保存的任务状态）重新投影成 model-visible reminder。

注意这和 DynamicContext 很像，但触发原因不同：

```text
DynamicContext:
  补当前日期/记忆这类外部上下文

Todo before_model:
  补 graph state 里还活着、但消息窗口里丢失的任务上下文
```

**ViewImage** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/view_image_middleware.py" sha="4a2f0a23" />，只有模型支持 vision 时才会装配。它把模型已经通过 `view_image` 工具读过的图片投影成 provider-compatible multimodal message，让后续 model call 真的能看到图片内容，而不只是看到“工具执行成功”。

这块要和工具本身分开看：

```text
view_image tool:
  读取图片，把图片信息写进 state["viewed_images"]
  返回 ToolMessage("Successfully read image")

ViewImageMiddleware:
  在 before_model 阶段读取 state["viewed_images"]
  构造 provider-compatible multimodal HumanMessage
```

也就是说，工具执行完成只代表 runtime 拿到了图片；模型下一轮能不能“看见图片”，还要靠 middleware 把图片投影进 model request。

<Design30>
ViewImage 暴露了一个通用问题：不要把“最后一条 HumanMessage”直接等同于“最后一次用户输入”。消息列表里可能有 dynamic context、todo reminder、summary、image projection 等 runtime 注入消息。3.0 需要给 runtime-injected messages 更明确的 provenance（来源、用途、幂等键），不要靠提示词文字承担控制元数据。
</Design30>

## wrap_model_call：包住模型调用

`wrap_model_call` 是最接近 model provider 的向内层。它拿到一个 `handler`，`handler(request)` 表示继续往内走，最终调用真实模型。调用 `handler` 前可以改 request；`handler` 返回后也可以处理 response；如果 `handler` 抛异常，则进入异常路径。

这一拍涉及的 middleware 比表面看起来多：

```text
ToolOutputBudget(model side)
DanglingToolCall
LLMErrorHandling
LoopDetection(wrap_model_call soft warning)
SkillActivation
Todo(wrap_model_call completion reminder)
DeferredToolFilter(model side)
```

**ToolOutputBudget** 在 model 侧做兜底修剪。它会检查历史里的超大 `ToolMessage`，把内容换成 head/tail preview 和外置路径，避免下一次 model call 被巨大工具结果拖垮。它不是摘要器，完整输出仍尽量保存在文件里。

这一点很容易和 Summarization 混淆：

```text
Summarization:
  conversation history boundary
  用摘要替换旧对话

ToolOutputBudget:
  single tool result boundary
  尽量外置完整原始输出，只把预览留在 ToolMessage
```

head/tail preview 也不是语义摘要。它只是导航信号：

```text
head:
  帮模型判断这是什么输出、结构长什么样

tail:
  帮模型看到结尾状态、错误、总结行

path:
  如果确实需要完整内容，再用 read_file 读取
```

所以它保护的是 model context budget，而不是替模型理解文件内容。

**DanglingToolCall** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/dangling_tool_call_middleware.py" sha="4a2f0a23" />，负责修补 provider 工具调用协议。它防的是这种历史：

```text
AIMessage(tool_calls=[call_1])
  ...缺少 ToolMessage(tool_call_id=call_1)
```

很多 provider 要求 `AIMessage.tool_calls[*].id` 后面必须有对应 `ToolMessage.tool_call_id`。DanglingToolCall 的算法是“先索引，再重建”：

```text
tool_messages_by_id:
  dict[tool_call_id, deque[ToolMessage]]

tool_call_ids:
  所有 AIMessage 声明过的 tool_call id

patched:
  重新构造后的本次 request.messages
```

第一步把已有 ToolMessage 按自己的 `tool_call_id` 分组：

```python
tool_messages_by_id[msg.tool_call_id].append(msg)
```

第二步遍历到 AIMessage 时，按它的 `tool_calls` 顺序查这个索引。查到真实 ToolMessage 就移动到 AIMessage 后面；查不到就补一条 synthetic `ToolMessage(status="error")`。所以顺序不是靠原消息里 ToolMessage 的位置保证，而是靠 `AIMessage.tool_calls[].id == ToolMessage.tool_call_id` 这个协议关系重建。

注意它修的是本次 `ModelRequest`，不是直接持久改 graph state：

```text
request.override(messages=patched)
```

这就是它选择 `wrap_model_call` 而不是 `before_model` 的原因：补丁必须插到对应 AIMessage 后面，不能简单 append 到末尾。

**LLMErrorHandling** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/llm_error_handling_middleware.py" sha="4a2f0a23" />。它守的是 model provider boundary：超时、连接失败、5xx、429、quota/auth 等异常在这里分类处理。可重试的 transient/busy 会退避重试；不可重试或重试耗尽，会返回 fallback `AIMessage`。

它处理的是 `handler(request)` 抛出的异常，也就是“没拿到正常 AIMessage”的情况。它不处理 provider 正常返回、但返回内容不可信的情况。这个边界很重要：下一站的 `SafetyFinishReasonMiddleware` 就是处理“provider 正常返回 AIMessage，但 finish_reason 表示安全截断，tool_calls 不可信”的情况。

它还有 circuit breaker（熔断器）：

```text
closed     电路连通，模型调用允许通过
open       电路断开，快速失败走 fallback
half_open  恢复探测，只放一个调用测试 provider 是否恢复
```

`closed` 在这里是健康态，不是“关闭请求”。这是行业术语，不是 DeerFlow 自己反着写。另一个关键点是 `GraphBubbleUp` 必须原样抛出，因为它代表 LangGraph 的中断/暂停/恢复控制流，不是普通异常。

**LoopDetection** 在 `wrap_model_call` 里主要注入 soft warning：如果上一轮 after_model 判定有循环风险，但还没到 hard stop，就把提醒插进下一次 model request。这样可以避免在 AIMessage(tool_calls) 和 ToolMessage 之间插入不合法消息。

**SkillActivation** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/skill_activation_middleware.py" sha="4a2f0a23" />。用户以 `/skill-name` 开头时，它会读取完整 `SKILL.md`，构造一条隐藏 `HumanMessage` 注入本次 request。这个内容不持久进入 graph state，属于 turn-local request overlay。这样 skill 只有被显式激活时才进入上下文，不会每轮污染 prompt。

它和 DynamicContext 的差别值得单独记：

```text
DynamicContext:
  before_agent
  写 graph state messages
  conversation-level context

SkillActivation:
  wrap_model_call
  只 override 当前 ModelRequest.messages
  turn-local context
```

SkillActivation 还会把 `SKILL.md` 包进 XML-like envelope：

```text
<slash_skill_activation>
  <user_request>...</user_request>
  <skill ...>
    <skill_content encoding="xml-escaped">...</skill_content>
  </skill>
</slash_skill_activation>
```

这里的 `xml-escaped` 指的是把 skill 内容里的 `<`、`>`、`&` 等字符转义成普通文本，避免 skill 文档里的标签样式内容破坏外层 prompt envelope。实现上用的是 Python 的 `html.escape()`。

如果 skill 不存在、被禁用、当前 agent 没权限使用，middleware 可以直接返回自己构造的 `AIMessage`，不浪费一次模型请求。

**Todo** 在 `wrap_model_call` 里负责消费 after_model 阶段排队的 completion reminder。如果模型试图在 todo 未完成时收尾，Todo 的 after_model 会先 jump 回 model；下一次 wrap_model_call 再把隐藏 reminder 注入本轮 request。

**DeferredToolFilter** 来自 <SourceRef file="backend/packages/harness/deerflow/agents/middlewares/deferred_tool_filter_middleware.py" sha="4a2f0a23" />。它在 model 侧隐藏还没被 `tool_search` promoted 的 deferred tool schema，避免一次性把大量 MCP 工具 schema 绑给模型。

<Design30>
DeferredToolFilter 的关键不是“藏 schema”本身，而是两点 enforcement：model 侧控制可见 schema，tool 侧控制真实可执行调用。只做前者不够，因为模型仍可能从历史或猜测中调用未 promoted 工具。3.0 可以把它抽象成一等的 `ToolVisibilityPolicy`，同时声明 visible、executable、blocked 三类工具。
</Design30>

<Philosophy title="工程设计哲学">

向内半程做的不是推理，而是准备模型调用：运行时状态就位，模型可见上下文就位，历史消息协议就位，request 里的 tool schema 和大输出也被整理好。

DeerFlow 没有把这些横切逻辑塞进 graph node，而是放进 middleware。好处是 graph 仍然保持“model ↔ tools”的主循环；代价是顺序变成隐形接口。一个 middleware 放错 hook，或者注册顺序不对，表面上代码能跑，实际语义可能已经变了。

</Philosophy>

<Tradeoff leftTitle="向内分层的好处" rightTitle="代价">
  <Fragment slot="left">

  - 横切准备逻辑不进 graph，主循环保持简单
  - lead 和 subagent 能复用 runtime 基座
  - 动态内容走 message stream，system prompt 保持稳定，利于 prefix cache

  </Fragment>
  <Fragment slot="right">

  - 顺序是隐形契约，主要靠注释和测试守住
  - 链按 run 条件变化，排查问题前要先还原本次装配结果
  - 同步 hook 如果做阻塞 IO，会卡住共享事件循环

  </Fragment>
</Tradeoff>

## 会绊倒你的地方

- **`before_agent` 只跑一次，`before_model` 每轮都跑。** 路径、上传、sandbox 生命周期属于 run 级准备；摘要、todo reminder、图片投影跟着每次 model call 走。
- **默认 lazy init 会改变直觉。** `ThreadData` 默认只计算路径，`Sandbox` 默认不在 before_agent 立刻 acquire。不要把“注册了 middleware”理解成“资源已经真实创建”。
- **链是按条件拼的。** plan mode、vision model、tool_search、subagent 开关、guardrail provider 都会影响实际 middleware 列表。先确认这条 run 里有没有该 middleware，再判断它为什么没生效。
- **`wrap_model_call` 不等于 `before_model`。** DanglingToolCall 必须把补丁插到历史正确位置，所以只能改本次 request；`before_model` 更适合返回 graph state update。
- **系统提示词是故意稳定的。** 日期、memory、skill、todo、图片这些动态信息都走 message/request overlay，不应该随手改 system prompt。

---

这一站只走到 model call。请求已经经过 `before_agent`、`before_model`、`wrap_model_call`，模型终于拿到整理后的 request。下一站走另一半：模型返回后，`after_model` 会按逆序接管结果，工具执行前后还会经过 `wrap_tool_call`，run 结束时再进入 `after_agent`。那一半才是安全策略、循环检测、工具边界和副作用真正密集发生的位置。
