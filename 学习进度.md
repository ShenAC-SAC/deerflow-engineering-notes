## 第一阶段：请求入口与 Agent 主链路

已经读完核心链路。

Embedded 入口：

```text
DeerFlowClient.stream()
  -> _get_runnable_config()
  -> _ensure_agent()
  -> create_agent()

```

Gateway 入口：

```text
POST /api/threads/{thread_id}/runs/stream
  -> stream_run()

  -> asyncio.create_task(run_agent(...))
  -> run_agent()
  -> make_lead_agent()
  -> create_agent()
  -> agent.astream()
  -> StreamBridge
  -> sse_consumer()
```

核心认知：

```text
Agent = model + tools + middleware + system_prompt + state_schema + checkpointer/store
```

已经讲清：

- DeerFlow 底层基于 LangGraph/LangChain agent runtime。
- `langgraph.json` 主要服务 LangGraph Server，不是 DeerFlow embedded/gateway 主链路的唯一入口。
- `make_lead_agent()` 是 DeerFlow 自己组装 lead agent 的工程入口。
- `create_agent()` 创建 LangChain/LangGraph agent graph，内部包含 agent loop。

## ThreadState 与 Middleware

已经进入核心阶段。

ThreadState 心智模型：

```text
ThreadState = LangGraph state schema + DeerFlow runtime 扩展字段
```

Reducer 心智模型：

```text
当多个节点 / middleware / tool 更新同一个字段时，
LangGraph 不直接裸覆盖，而是根据字段上的 reducer 合并。
```

`Command(update=...)`：

```text
是节点/middleware/tool 返回给 LangGraph 的状态更新指令。
具体怎么合并，由 state schema 上的 reducer 决定。
Command 里不用指定 reducer。
```

当前最重要的 middleware 心智模型：

```text
middleware 不是一个统一意义上的 runtime。
它们分布在 model 前、model 后、tool 执行前后。
每个 middleware 的 hook 决定它能控制什么，不能控制什么。
```

## 已详细学习的 Middleware

### ThreadDataMiddleware

- 注入 thread/user/workspace 相关上下文。
- 给后续 middleware/tools 提供运行期信息。

### UploadsMiddleware

- 处理用户上传文件。
- 用户上传可能进入 message，也可能作为 attachment/file metadata 被 middleware 投影给模型。
- `wrap_model_call(request, handler)` 里的 request 不是 HTTP request，而是 LangChain model request。
- `handler` 是 LangChain 内部传入的下游调用函数。

### MemoryMiddleware

- 使用 memory queue 异步处理对话记忆。
- queue 的价值是解耦主 run 和记忆更新，不阻塞用户响应。
- `get_memory_queue()` 是全局 singleton，用 lock 保证多线程下只初始化一次。

### DeerFlowSummarizationMiddleware

- `before_summarization` 是一组可调用 hook。
- `Protocol` 是结构化类型约束：不要求继承，只要形状匹配。
- `if not []` 为 True，所以空 hook 列表会直接 return。

### TodoMiddleware

- `before_model` 注入隐藏的 `HumanMessage(name="todo_reminder")`。
- `after_model` 可以在未完成 todo 且模型准备结束时跳回 model。
- `wrap_model_call` 可以把 pending reminder 注入 request messages。
- `write_todos` 出现在消息中，说明模型已经处理 todo，不需要重复提醒。
- `aafter_model` / `abefore_model` 是 async hook 命名，前面的 `a` 表示 async。

### ViewImageMiddleware

- `view_image_tool` 读取图片，写入 `state["viewed_images"]`。
- 工具返回 `ToolMessage("Successfully read image")` 告诉模型工具调用已完成。
- middleware 再把图片投影成 provider-compatible multimodal `HumanMessage`。
- `mime_type` 是 `image/png` / `image/jpeg` 这种内容类型。
- `HumanMessage` 用来承载给模型看的输入，不一定真的是用户亲手发的消息。
- 隐藏字段用于前端不展示 runtime 注入消息。

### DeferredToolFilterMiddleware

- 控制工具是否暴露给模型。
- `promoted` 是 DeerFlow 自己的 state 字段，不是 LangGraph 内置。
- model 不直接理解 `promoted`，middleware 根据它修改 `request.tools`。
- 执行层也要校验，避免模型调用未 promoted 的工具。

### SubagentLimitMiddleware

- 在 `after_model` 阶段检查模型刚输出的 `task` tool calls。
- 只看 `messages[-1]`，因为 `after_model` 发生在 model node 后、tools node 前。
- 会裁剪多余 subagent task。
- 保留 message id，让 reducer 替换原 AIMessage，而不是追加一条。
- `MIN_SUBAGENT_LIMIT=2` 是配置下限，不表示模型不能只启动一个 subagent。
- 当前 subagent 任务划分主要靠 prompt 和模型能力，没有独立 task-split judge。

### LoopDetectionMiddleware

- 在 `after_model` 检测重复工具调用。
- 检测方式：
  - hash-based：同一组 tool calls 重复。
  - frequency-based：同类工具调用过多。
- `read_file` 用 200 行 bucket 做稳定 key。
- `json.dumps(..., sort_keys=True, default=str)` 用来生成稳定可比较字符串。
- `normalized.sort()` 让工具调用顺序不影响 hash。
- soft warning 会等下一次 `wrap_model_call` 注入，避免破坏 AIMessage -> ToolMessage 协议。
- hard stop 会清空 tool_calls 并改写 AIMessage 内容。

## 当前学习位置：Middleware 模块已完成，下一步做总结复盘

ClarificationMiddleware 已完成一轮核对并沉淀笔记：

```text
backend/packages/harness/deerflow/tools/builtins/clarification_tool.py
backend/packages/harness/deerflow/agents/middlewares/clarification_middleware.py
backend/packages/harness/deerflow/agents/lead_agent/prompt.py
```

关键事实：

1. `ask_clarification_tool` 定义为：

```python
@tool("ask_clarification", parse_docstring=True, return_direct=True)
```

2. 它主要暴露给模型工具 schema，真实逻辑由 `ClarificationMiddleware` 处理。

3. middleware 的核心逻辑：

```python
if request.tool_call.get("name") != "ask_clarification":
    return handler(request)

return Command(update={"messages": [tool_message]}, goto=END)
```

4. 它是单个 tool-call 级别的拦截器，不是整轮 `AIMessage.tool_calls` 仲裁器。

5. `ToolMessage` 是 runtime 构造的工具结果，用来匹配原始 `tool_call_id`。

6. 前端会识别：

```ts
message.type === "tool" && message.name === "ask_clarification"
```

然后展示成 clarification UI。

### 重要纠正：ask_clarification 的正常路径和异常混合路径

正常设计期望：

```text
model -> 只调用 ask_clarification
tools -> middleware 生成 ask ToolMessage
frontend/channel 识别并展示问题
run 等用户响应
```

DeerFlow prompt 明确要求：

```text
Clarify -> Plan -> Act
ask_clarification 必须 FIRST
不要开始工作后再澄清
```

但如果模型异常地生成混合工具调用：

```text
[A, ask_clarification, C]
```

当前 runtime 没有在整轮 `tool_calls` 层面强制禁止。

本地最小实验看到：

```text
A 会执行
C 会执行
ask_clarification 会生成 ToolMessage
然后可能继续回 model
```

重要结论：

```text
ask_clarification 的 question/context/options 是模型在工具执行前生成的。
所以它不可能基于同一轮 A/C 的工具返回结果。
A/C 的结果只可能影响下一次 model 调用。
```

还有一个重要修正：

```text
Command(goto=END) 不是混合工具场景下的全局立即刹车。
```

在 `create_agent + ToolNode` 的组合里，多个 tool calls 可能作为多个分支执行并合并。`return_direct=True` 能让“只有 ask_clarification”的场景结束，但混入普通工具时，不保证立即结束。

本轮新确认：

```text
goto=END 是返回给 LangGraph graph runner 的 Command 控制指令。
它不是传给 DeerFlow run_agent() 的普通返回值，也不是 ToolNode 内部的同步取消信号。
```

本地依赖源码与最小实验确认：

```text
ToolNode async 路径会用 asyncio.gather(*coros) 执行同一 AIMessage 的多个 tool calls。
所以混合 [normal_tool, ask_clarification] 时，普通工具可能已经执行。
```

`create_agent` 的 tools-to-model 路由规则是：

```text
如果所有已执行 client-side tools 都是 return_direct=True:
  route to END
否则:
  route back to model
```

因此最终结论：

```text
only ask_clarification
  -> return_direct + Command(goto=END)
  -> 本轮 run 结束，不回 model

ask_clarification mixed with normal tools
  -> sibling tools may execute
  -> not all tools are return_direct
  -> graph may return to model
```

## 已完成学习：SandboxMiddleware

已讲清第一层心智模型：

```text
ThreadDataMiddleware -> 文件路径上下文 / thread file space
SandboxMiddleware    -> 工具执行环境上下文 / execution space
```

`SandboxMiddleware` 不执行命令。它负责准备、复用、持久化工具执行环境身份：

```text
state["sandbox"] = {"sandbox_id": "..."}
```

默认 `lazy_init=True`：

```text
不是 run 一开始就创建 sandbox
而是等第一个真正需要 sandbox 的 tool call 再创建
```

关键状态语义：

```text
工具内部直接 mutate runtime.state["sandbox"]
  不等于 LangGraph graph state 已经持久更新

跨节点可见 / checkpoint-resume 可见的状态
  需要通过 Command(update={"sandbox": ...}) 返回
```

所以 `SandboxMiddleware.wrap_tool_call()` 会比较：

```text
prev_sandbox_id before handler(request)
curr_sandbox_id after handler(request)
```

如果发现 sandbox 是工具调用过程中 lazy 创建的，就把工具结果包装或合并成：

```python
Command(update={
    "sandbox": {"sandbox_id": sandbox_id},
    "messages": [tool_message],
})
```

### Sandbox + Subagent + Memory 的 3.0 设计锚点

本轮讨论确认：

```text
共享 sandbox / thread_data
  不等于
共享 semantic memory
```

DeerFlow 当前 `task` tool 启动 subagent 时会传入 parent context：

```text
sandbox_state
thread_data
thread_id
```

所以 subagent 不是完全 cold start，它能继承执行环境和线程文件上下文。但它还没有一个实时的、结构化的 sibling-subagent shared memory。

适合 DeerFlow 3.0 的方向：

```text
run/thread-scoped shared structured findings
```

也就是让并行 subagents 共享证据化发现，而不是共享原始上下文：

```text
scope
kind
claim
evidence
appliesTo
confidence
producedBy
invalidatesWhen
```

### Sandbox 执行链路最终结论

`runtime.state["sandbox"]` 是 graph state 实例里的 sandbox 子状态：

```python
runtime.state["sandbox"] = {"sandbox_id": "local:thread-123"}
```

`ThreadState` 是这个 graph state 的 schema/type 定义；`ToolRuntime.state` 是当前工具调用拿到的具体 graph state 实例。

工具执行不是 ToolNode 自动“进入 sandbox”。ToolNode 只负责调用工具并注入 `runtime`。真正链路是：

```text
tool function
  -> ensure_sandbox_initialized(runtime)
  -> runtime.state["sandbox"]["sandbox_id"]
  -> get_sandbox_provider().get(sandbox_id)
  -> Sandbox instance
  -> sandbox.execute_command/read_file/write_file/list_dir
```

`bash_tool(command="...")` 里的 `command` 是模型生成的 tool-call 参数，位于 `AIMessage.tool_calls[].args.command`，不是独立 LangGraph state 字段，也没有自己的 reducer。

`provider` 可以理解成“后端能力实现 / backend adapter”。这里的 sandbox provider 不是 model provider：

```text
model provider   -> 模型调用能力
sandbox provider -> 工具执行环境能力
```

LocalSandboxProvider：

```text
本地文件系统 / 本机执行适配器
通过 path mappings 把 /mnt/user-data/... 映射到真实 thread workspace
bash 使用 subprocess.run([shell, "-c", command])
不会打开可见交互终端
host bash 是主机 shell 执行权限，不是网络权限
不是强安全隔离
```

container-based AioSandboxProvider：

```text
通过 config.yaml 配置 sandbox.use: deerflow.community.aio_sandbox:AioSandboxProvider
可以走 Docker/DooD：gateway 访问 Docker daemon 创建 sandbox container
也可以走 provisioner_url：gateway 请求 provisioner 创建 Kubernetes Pod
具体 sandbox 运行在哪里，取决于 gateway 所在环境和 provider 能访问的 Docker/provisioner
```

路径虚拟化作用：

```text
给模型稳定路径：/mnt/user-data/workspace/uploads/outputs
统一 local/container/pod 的工具接口
隐藏宿主机真实路径
便于输出脱敏
但虚拟路径本身不是安全边界
```
